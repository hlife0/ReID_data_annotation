# 需求文档 E：全局队列分发与前端预缓存优化（设计稿）

## 1. 目标

在**不改变现有 SQLite 数据库结构**的前提下，优化 UI 标注体验，重点降低用户点击“跳过当前帧”或“提交并下一帧”后的等待时间。

本次需求的核心目标：

1. 后端不再对每次请求都临时从候选集合里独立抽样，而是在内存中维护一个**全局候选队列**。
2. 队列按分层桶生成，优先级固定为：
   - `annotation_count = 0`
   - `annotation_count = 1`
   - `annotation_count = 2`
   - 若以上均空，则退化为“全局最小 count 桶”
3. 当帧被分配出去后，无论用户是“跳过”还是“提交”，该帧都会先从当前轮队列中移除，不立即回到候选头部。
4. 前端在拿到真实分配帧后，同时预取未来 5 帧的图片和 AI 框信息。
5. 前端预取只是缓存优化，**真实下一帧永远以后端返回为准**。

本需求的重点是**体验优化**，不是严格意义上的 per-annotator deterministic 分配。

---

## 2. 范围与约束

### 2.1 必须保持不变的内容

1. SQLite 表结构不变。
2. 现有批次目录结构不变。
3. 现有导出文件结构不变：
   - `reviewed/*.reviewed.csv`
   - `reviewed_raw/*.frame_records.jsonl`
4. 现有提交流程不变：
   - 写入 `annotations`
   - 更新 `track_person_stats`
   - 更新 `frame_counts`
   - 分配下一帧
5. 前端不负责真实选帧，前端缓存只是优化层。

### 2.2 明确不做的事情

1. 不新增数据库表或字段。
2. 不引入“某帧被某人领取后暂时锁住不给别人”的排他分配。
3. 不要求严格 deterministic 到“同一个 annotator 的未来序列可完全复现”。
4. 不依赖浏览器 HTTP 缓存作为正确性基础。
5. 不改管理后台统计口径。

---

## 3. 方案概览

总体方案分两层：

1. **后端真实分配层**
   - 在服务端内存维护一个“当前活跃桶”的候选队列。
   - 每次分配都从队列头取帧。
   - 队列空了或桶变化了，再整体重建。

2. **前端体验优化层**
   - 后端在返回真实分配帧时，同时返回“未来 5 帧预取线索”。
   - 前端在后台下载这 5 帧图片并缓存到 JS 内存。
   - 用户下一次点击时，如果真实分配帧命中缓存，就立即显示；未命中则正常回源下载。

因此：

- 队列决定“分配体验”
- 预取决定“等待体验”
- 数据库仍然是唯一事实来源

---

## 4. 后端分发队列设计

### 4.1 队列的性质

后端新增一个**仅存在于内存中的全局队列状态**，用于优化候选帧调度。

该状态不是事实存储，不需要落盘。

建议包含以下字段：

1. `active_bucket_count`
   - 当前活跃桶编号，例如 `0 / 1 / 2 / min_count_fallback`
2. `queue`
   - 当前活跃桶的待分配帧队列，元素为 `(video_stem, frame_index)`
3. `generation`
   - 队列重建次数，用于日志与调试
4. `seed`
   - 服务启动时已有的随机 seed，可继续复用

说明：

- 队列是**全局共享**的，不按 annotator 拆分。
- 队列是**优化状态**，可以在服务重启后丢失。

### 4.2 活跃桶选择规则

每次需要重建队列时，按如下顺序选择活跃桶：

1. 若存在 `annotation_count = 0` 的帧，则活跃桶为 `0 次桶`
2. 否则，若存在 `annotation_count = 1` 的帧，则活跃桶为 `1 次桶`
3. 否则，若存在 `annotation_count = 2` 的帧，则活跃桶为 `2 次桶`
4. 否则，活跃桶退化为“全局最小 `annotation_count` 桶”

该规则保持了明确的分层优先级。

### 4.3 队列重建逻辑

当出现以下任一情况时，重建队列：

1. 当前队列为空
2. 当前活跃桶已经变化
3. 队列中的帧大量失效并被清空

重建步骤：

1. 查询当前活跃桶
2. 取出该桶中全部候选帧
3. 在 Python 内存中打乱顺序
4. 生成新的队列
5. `generation + 1`

打乱方式要求：

- 不必追求严格 deterministic
- 可以使用现有 `random.Random(seed)` 派生的打乱方式
- 只要求“用户体感不是固定顺序”即可

### 4.4 出队与复验逻辑

每次真实分配时：

1. 在锁内检查队列是否需要重建
2. 从队列头部取出一个候选帧
3. 回表检查该帧当前是否仍属于活跃桶
4. 若已失效，则丢弃并继续取下一个
5. 若有效，则作为本次真实分配结果
6. 记录 `assignments`

这一步“回表复验”是必须的，因为：

- 其他请求可能已经提交并改变了某些帧的 `annotation_count`
- 队列是内存快照，不保证与数据库状态永久一致

### 4.5 跳过语义

当用户点击“跳过当前帧”时：

1. 当前帧不会修改 `annotation_count`
2. 下一帧从当前队列继续取
3. 刚才被跳过的帧**不会立即回到当前轮队列**
4. 只有在未来某次队列重建时，该帧才可能重新进入候选队列

这样做的目的：

- 避免用户刚跳过的帧立刻再次出现
- 改善体感

### 4.6 提交语义

当用户点击“提交并下一帧”时：

1. 按现有逻辑写入 annotation
2. 将当前帧 `annotation_count + 1`
3. 然后再基于最新数据库状态从队列分配下一帧

若当前提交导致活跃桶变化，则在下一次分配前触发队列重建。

## 5. 前端预缓存设计

### 5.1 预缓存目标

前端不参与决定真实下一帧，只做缓存优化。

每次获取真实帧后，前端同时后台预取未来 5 帧的图片资源，以便下一次真实命中时能快速展示。

### 5.2 缓存对象

前端建议维护 JS 内存缓存，而不是依赖浏览器 HTTP 缓存。

原因：

1. 当前图片接口返回 `Cache-Control: no-store`
2. 当前前端图片请求还会附带时间戳参数 `_ts`
3. 因此浏览器默认缓存策略并不适合直接复用

建议前端维护两个 Map：

1. `imageCache`
   - key: `video_stem + frame_index`
   - value: 已下载图片的 `blobUrl` 或 `ImageBitmap`
2. `inflightFetches`
   - key: `video_stem + frame_index`
   - value: 当前正在预取的 Promise

### 5.3 预取窗口

建议采用**滑动窗口**，而不是每次无脑重拉 5 帧。

策略如下：

1. 后端返回 5 个预取候选
2. 前端只对未缓存、未在下载中的帧发起请求
3. 当用户消费掉当前帧后，再用新响应中的候选更新窗口
4. 缓存总量建议略大于 5，例如保留 8 到 12 帧，避免频繁抖动

### 5.4 命中与失效

当真实下一帧到达时：

1. 如果图片已在前端缓存中，直接展示
2. 如果未命中，则走正常加载流程
3. 如果候选图片因并发变化已经“预测错了”，也不影响正确性

因此：

- 预取命中时，用户看到的是“几乎瞬时切换”
- 未命中时，行为退化回当前系统，不影响可用性

---

## 6. 后端返回的预取线索

### 6.1 设计原则

后端在返回真实分配帧时，同时附带“未来 5 帧预取线索”。

这些线索只是**缓存提示**，不是任务承诺。

### 6.2 建议返回字段

对于 `POST /api/next_frame`，返回结构可扩展为：

```json
{
  "ok": true,
  "frame": { ...真实分配帧... },
  "prefetch_frames": [
    {
      "video_stem": "20260211_171423",
      "frame_index": 101,
      "timestamp_ms": 1770801270000.0,
      "image_url": "/api/frame_image?video_stem=20260211_171423&frame_index=101",
      "ai_boxes": [ ... ],
      "image_width": 640,
      "image_height": 480
    }
  ]
}
```

对于 `POST /api/submit`，返回结构可扩展为：

```json
{
  "ok": true,
  "submitted": { ... },
  "next_frame": { ...真实下一帧... },
  "prefetch_frames": [ ... ]
}
```

### 6.3 `prefetch_frames` 的生成原则

生成方式建议为：

1. 先确定真实当前分配帧
2. 再基于当前队列剩余内容，选取后续最多 5 个有效候选帧
3. 返回这些帧的轻量元信息

说明：

- 这些帧不需要写入 `assignments`
- 这些帧不算正式分配
- 前端只能把它们当作缓存提示

---

## 7. 现有数据库与新方案的关系

### 7.1 数据库仍然是唯一事实来源

虽然新增了后端内存队列和前端预缓存，但以下事实仍由数据库决定：

1. 哪些帧合法存在：`frames`
2. 当前每帧已标次数：`frame_counts`
3. 哪些帧已被提交：`annotations`
4. 分配日志：`assignments`
5. AI 轨迹统计：`track_person_stats`

### 7.2 不新增任何落盘结构

本需求下不需要新增：

1. 新的 SQLite 表
2. 新的 SQLite 列
3. 新的 JSON 队列文件
4. 新的磁盘缓存索引文件

### 7.3 可选的内存状态

允许新增但不要求落盘的只有：

1. 后端全局候选队列
2. 前端图片缓存 Map
3. 前端下载中的 Promise Map

---

## 8. 并发语义

### 8.1 后端并发要求

后端队列的所有读写都必须放在现有分配锁内执行。

即：

1. 队列检查
2. 队列重建
3. 队列出队
4. 活跃桶复验
5. assignment 写入

必须处于同一串行控制路径中。

### 8.2 多用户并发下的行为

该方案不保证“不同 annotator 永远不会拿到同一帧”。

但它能保证：

1. 单进程内不会因为队列竞争而把**同一个队列头**同时发给两个请求
2. 跳过的帧不会立刻回到当前轮队列头部
3. 预取失效不会影响真实分配正确性

### 8.3 预取与真实分配的关系

必须明确：

- `prefetch_frames` 是预测缓存，不是保留名额
- 即使前端已经预取了 5 张图，下一次真实分配也仍然后端说了算
- 若真实分配与预取不一致，前端直接使用真实结果

---

## 9. 性能判断

### 9.1 CPU

按当前项目规模，该方案 CPU 可接受。

原因：

1. 当前帧总量约为数千级
2. 当前分配频率是“人工点击触发”，不是高频自动流量
3. 真正耗时的大头仍是视频帧读取与 JPEG 编码，而不是队列出队逻辑

### 9.2 内存

新增内存消耗主要来自：

1. 后端候选队列
   - 只存 frame key，成本很低
2. 前端图片缓存
   - 5 到 12 张 JPEG 的量级，可接受

### 9.3 带宽与请求量

预取会增加图片请求量，但命中时能明显降低等待感。

为了避免浪费，必须：

1. 做去重下载
2. 做滑动窗口
3. 限制缓存上限

## 11. 验收标准

满足以下条件即视为达标：

1. 当存在 `annotation_count = 0` 的帧时，只从 `0 次桶` 中出队。
2. 当 `0 次桶` 为空且存在 `annotation_count = 1` 的帧时，只从 `1 次桶` 中出队。
3. 当 `0 次桶`、`1 次桶` 都为空且存在 `annotation_count = 2` 的帧时，只从 `2 次桶` 中出队。
4. 用户连续点击“跳过当前帧”时，不会立即再次拿到刚刚跳过的帧。
5. 后端真实分配结果仍写入现有 `assignments`，不新增 schema。
6. 前端即使命中失败，也不会影响真实分配正确性。
7. 命中预取缓存时，下一帧显示等待时间明显缩短。
8. 服务重启后，系统仍可正常工作，只是队列与前端缓存会重新开始。

---

## 12. 风险与备注

### 12.1 主要风险

1. 若前端缓存处理不当，可能重复下载同一帧图片。
2. 若后端不做回表复验，可能出现队列与数据库状态漂移。
3. 若队列操作不放在现有锁内，可能导致并发下顺序混乱。

### 12.2 取舍说明

本方案的核心取舍是：

- 放弃“严格 deterministic 分配”
- 换取“更简单的实现”和“更好的体感性能”

这符合当前目标，即优先提升标注体验。

---

## 13. 总结

该方案的本质是：

1. 用后端内存队列减少“重复抽中刚跳过帧”的问题
2. 用前端 5 帧预缓存减少“下一帧等待图片加载”的问题
3. 用数据库继续承担唯一事实来源

因此，它可以在**不改数据库结构**的前提下，较低风险地提升标注体验。
