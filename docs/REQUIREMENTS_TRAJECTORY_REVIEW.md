# 需求文档 X：轨迹级问题驱动标注提效规范（规划与执行跟踪版）

最后更新：2026-04-15（R0 第一轮收稳完成）

## 1. 文档定位

本文件不是替代现有 A/B/C/F/G/H 各阶段需求文档，而是补充一份跨阶段的“标注提效总文档”。

它固定回答三件事：

1. 理想状态是什么
2. 具体应该怎么一步步落地
3. 当前已经做到哪里，下一步该做什么

适用范围：

- 现有 `annotation/` 批次上的 AI 预标注 + 人工复核流程
- 现有 `codex/ui_review_server.py` / `codex/ui_admin_server.py` / `codex/ui_review_web/*` / `codex/ui_admin_web/*`
- 当前已经支持的“按原始 IMU 段进入 annotation 主流程”的长视频任务形态

本文件默认以 2026-04-10 新采集批次为第一落地对象，但设计目标不局限于该批次。

## 2. 背景与问题定义

### 2.1 当前项目现状

当前仓库已经具备以下基础能力：

1. 能把连续长视频按原始 IMU 记录段整理成 annotation 可消费的 session。
2. 能对这些 session 跑 A 阶段 AI 预标注，输出逐帧 `pseudo_labels/*.auto.csv`。
3. 能启动 review/admin 服务并进行人工复核。
4. 当前新正式批次为 `./annotation/batch_20260413_v01`。

对这批新数据，当前已知基线是：

1. 任务单位：按原始 IMU 段构造的 20 个 session。
2. AI 预标注：已完成，生成 20 个 `.auto.csv`，总伪标注行数约 423048。
3. review 底层：已能动态加载新 batch，不再只认旧的 4 个视频。
4. 槽位方向：已开始从固定 `P1/P2` 向动态 `P1-P7` 迁移，但仍是原型状态。

### 2.2 当前真正的效率瓶颈

当前最主要的问题不是“AI 预标注还不够强”，而是人工工作流仍然近似按“逐帧图片标注”在运行。

这会带来五类浪费：

1. 工作单位错位：同一轨迹在连续几十帧都稳定，标注员仍然需要反复确认。
2. 绝大多数时间花在“确认没问题”，不是“修真正有问题的地方”。
3. 多目标场景中，切换对象、切换帧、重新找同一个人的成本很高。
4. 当前任务边界主要按数据管理切，不是按人类连续处理效率切。
5. AI 现在更多只是“给起点”，还没有真正承担“过滤稳定帧、只把问题交给人”的角色。

### 2.3 本文档的核心判断

本项目下一阶段最值得投入的方向不是继续优化逐帧修框细节，而是把人工工作单位改成：

1. 轨迹级工作单位
2. 关键帧修正 + 局部自动传播
3. 风险片段 / 问题点驱动复核

也就是说，默认工作流应从：

- “给我下一帧”

转成：

- “给我下一段需要处理的轨迹问题”

## 3. 理想状态

### 3.1 人工工作单位

理想状态下，人工不再以“单帧”为默认操作对象，而是以“一个 session 内的一段问题区间”作为工作单位。

一个人工任务应包含：

1. `video_stem`
2. `issue_id`
3. 问题时间窗 / 帧区间
4. 涉及的一个或多个 AI `track_id`
5. 当前可见的动态槽位集合（通常为 `P1-P7`，但不写死上限）
6. 风险原因列表

人工进入任务后，不应该从第 1 帧机械往后翻，而应该：

1. 先看整个片段的轨迹概况
2. 直接跳到高风险点
3. 只修关键帧
4. 允许把修改传播到相邻稳定区间
5. 再跳去下一个问题点

### 3.2 UI 理想状态

理想状态下，review UI 应满足以下原则：

1. 左侧主画布保持大，不因为右侧信息增多而明显缩小。
2. 右侧不再是 7 套完整编辑面板堆叠，而是“对象卡片列表 + 单个共享编辑器”。
3. 底部或主区域附近有时间轴/风险点视图，用于快速跳转。
4. 存在“一键下一问题点”而不是“一键下一帧”。
5. 当前选中的对象在切帧或跳点后保持锁定。
6. 常用操作都应支持键盘优先。

右侧信息布局建议为：

1. 顶部：当前 session / 当前 issue / 当前风险类型
2. 中部：对象卡片列表
3. 下部：共享 bbox 编辑器与区间级操作按钮

对象卡片至少应支持：

1. 当前是否可见
2. 当前是否高风险
3. AI `track_id`
4. 最近是否被人工修改
5. 一键显隐 / 锁定 / 跳转

区间级操作至少应支持：

1. 接受本段 AI 结果
2. 当前对象本段 `occluded`
3. 当前对象本段 `outside`
4. 删除整段误检
5. 从当前位置 `split`
6. 与另一段 `merge`

### 3.3 后端理想状态

理想状态下，review 后端应该同时维护两套视图：

1. 轨迹/问题视图：给人工派单和交互使用
2. 逐帧视图：给历史兼容导出和下游流程使用

也就是说，前台按轨迹级工作，后台仍能导出逐帧结果。

后端应新增的核心中间产物包括：

1. `track_summary`
2. `risk_spans`
3. `issue_pool`
4. `keyframe_edits`
5. `propagated_frame_records`

其中：

1. `track_summary` 负责描述一个 session 内的 AI 轨迹结构。
2. `risk_spans` 负责描述哪些时间区间值得优先人工查看。
3. `issue_pool` 负责成为新的派单单位。
4. `keyframe_edits` 记录人工只在关键帧做过哪些修改。
5. `propagated_frame_records` 负责把关键帧结果扩展回逐帧输出。

### 3.4 风险驱动理想状态

系统不应把所有帧平均对待，而应先给每个轨迹/区间打风险分。

第一版风险信号可采用启发式规则，不要求复杂模型：

1. AI 置信度低
2. 相邻帧 bbox 变化异常大
3. 多个框重叠突然变高
4. 轨迹出现 / 消失
5. 目标数发生变化
6. 可能的 ID switch / split / merge
7. 遮挡持续时间长
8. IMU 数量或活跃设备集合变化
9. 视频时间窗接近原始 IMU 段边界

理想派单方式应分层：

1. 绿色：稳定段，默认通过，只抽样 QA
2. 黄色：中风险段，只看关键帧
3. 红色：高风险段，人工重点处理

### 3.5 IMU 在人工阶段的理想角色

IMU 在人工标注阶段应主要承担“提示与裁决证据”角色，而不是主操作区常驻大面板。

默认只展示轻量信息：

1. 当前段活跃 IMU 数量
2. 当前段设备列表
3. 同步状态
4. 绑定置信度或冲突提示

只有在这些情况才展开更深的 IMU 辅助信息：

1. 轨迹身份冲突
2. 目标数与设备数明显不一致
3. IMU 活动变化和视觉轨迹变化明显矛盾

### 3.6 与现有系统的兼容要求

理想状态不应推翻现有链路，而应满足：

1. 继续兼容当前 `pseudo_labels/*.auto.csv`
2. 继续兼容当前批次目录结构
3. 继续能导出 `reviewed_raw/*.jsonl` 与 `reviewed/*.csv`
4. 允许新旧两种 review 模式并行一段时间
5. 允许先在单个 batch 上试点，再逐步推广

## 4. 具体实行步骤

本节按“先高 ROI、再深改造”的顺序给出实施路线。

### 4.1 阶段 R0：基线稳定化

#### 目标

把当前“新批次 + 动态 session + 多 IMU + P1-P7 原型”的现状收稳，作为后续提效改造的基线。

#### 主要工作

1. 固化正式 batch 路径与运行方式。
2. 固化 review/admin 服务对动态 batch 的加载方式。
3. 完成 `P1-P7` 动态槽位从原型到可用版的最小清理。
4. 保持左侧主画布大、右侧控制区压缩。

#### 主要代码位置

1. `./codex/ui_review_server.py`
2. `./codex/ui_admin_server.py`
3. `./codex/ui_review_web/index.html`
4. `./codex/ui_review_web/app.js`
5. `./codex/ui_review_web/styles.css`
6. `./codex/ui_admin_web/app.js`

#### 交付标准

1. 新 batch 可稳定启动 review/admin 服务。
2. 新 batch 可通过 ngrok 稳定访问。
3. UI 不再依赖固定 `P1/P2` 字段。
4. 至少在当前批次上可以正常浏览、编辑、提交。

### 4.2 阶段 R1：离线生成轨迹摘要与风险摘要

#### 目标

在不改人工入口的前提下，先把“问题在哪里”离线算出来。

#### 主要工作

新增一条离线分析链，输入每个 session 的 `.auto.csv`，输出：

1. 每条轨迹的覆盖范围、关键属性和统计
2. 每个 session 的风险点和风险区间
3. 可直接供 review 派单使用的 `issue_pool`

#### 建议新增产物

```text
./annotation/batch_<...>/
├── review_prep/
│   ├── <video_stem>.track_summary.json
│   ├── <video_stem>.risk_spans.json
│   └── <video_stem>.issue_pool.csv
```

#### 主要代码位置

建议新增：

1. `./codex/process_review_issue_prep.py`

可能复用：

1. `./codex/process_prelabel_batch.py`
2. `./codex/ui_review_server.py`

#### 第一版风险规则建议

1. `score` 低于阈值
2. 同一 `track_id` 相邻帧位移或尺寸突变
3. 同帧多个框高重叠
4. 新轨迹出生 / 老轨迹消失
5. 人数变化
6. 靠近原始 IMU 段边界

#### 交付标准

1. 每个 session 都能生成轨迹摘要和风险摘要。
2. 能从批次层面统计红/黄/绿问题量。
3. 不改现有 review 流程也能先离线查看这些摘要。

### 4.3 阶段 R2：review 后端从“帧池”升级为“问题池”

#### 目标

让 review 服务先学会分配“问题片段”，而不是只分配“下一帧”。

#### 主要工作

1. 在保留旧 frame 模式的同时，引入 issue 模式。
2. review server 从 `frame_pool` 扩展到 `issue_pool`。
3. 支持“下一问题点 / 下一问题片段”的 API。
4. 保留导出为逐帧结果的兼容能力。

#### 建议接口

1. `POST /api/next_issue`
2. `GET /api/issue_detail?...`
3. `POST /api/save_issue`

旧接口可暂时保留：

1. `POST /api/next_frame`
2. `POST /api/submit`

#### 主要代码位置

1. `./codex/ui_review_server.py`
2. `./codex/ui_admin_server.py`

#### 交付标准

1. 后端能同时支撑 frame 模式和 issue 模式。
2. 一个 issue 可以对应某个 session 的一个风险区间。
3. 派单不再只能随机给帧。

### 4.4 阶段 R3：review 前端从“逐帧修框”升级为“问题点工作台”

#### 目标

把前端主交互改成“对象列表 + 风险跳转 + 共享编辑器 + 下一问题点”。

#### 主要工作

1. 右侧压缩成动态对象卡片列表。
2. 引入风险点跳转和当前对象锁定。
3. 增加底部或主区域的时间轴/风险标记。
4. 加入“一键下一问题点”。
5. 所有 UI 默认基于动态槽位，不再有固定双人假设。

#### 主要代码位置

1. `./codex/ui_review_web/index.html`
2. `./codex/ui_review_web/app.js`
3. `./codex/ui_review_web/styles.css`

#### 交付标准

1. 左侧主画布大小与当前版本基本一致。
2. 右侧在 `P1-P7` 甚至更多槽位下仍不拥挤。
3. 用户能明显减少“切帧找问题”的时间。

### 4.5 阶段 R4：关键帧修正 + 局部自动传播

#### 目标

让一次人工修改影响一段连续区间，而不是只影响当前帧。

#### 主要工作

1. 人工只在关键帧修框。
2. 系统对相邻稳定区间做局部传播或插值。
3. 支持轨迹级 `split / merge / reappear / occluded / outside` 操作。
4. 传播结果在导出时恢复为逐帧记录。

#### 主要代码位置

建议新增：

1. `./codex/review_propagation.py`

修改：

1. `./codex/ui_review_server.py`
2. `./codex/ui_review_web/app.js`

#### 交付标准

1. 人工一次编辑能覆盖一段连续时间窗。
2. 导出结果仍保持逐帧格式。
3. 当前批次上能明显减少重复修框次数。

### 4.6 阶段 R5：风险分层 QA 与自动通过策略

#### 目标

让稳定段尽量不进入人工视野，只对少量样本做抽查。

#### 主要工作

1. 建立红/黄/绿三层风险策略。
2. 绿色段默认通过，只抽样 QA。
3. 黄色段只看关键帧。
4. 红色段进入重点人工处理。
5. 后台增加质量抽样和统计。

#### 主要代码位置

1. `./codex/ui_admin_server.py`
2. `./codex/ui_admin_web/app.js`
3. `./codex/process_review_issue_prep.py`

#### 交付标准

1. review 工作量不再与总帧数线性相关。
2. QA 统计能独立监控质量。

### 4.7 阶段 R6：任务切分从“数据边界”扩展到“人类效率边界”

#### 目标

在保留原始 IMU 段作为数据管理边界的前提下，为人工派单再生成更合适的任务边界。

#### 主要工作

1. 将“数据分段”和“人工任务分段”解耦。
2. 根据复杂度和风险密度决定是否合并/拆分。
3. 在任务边界保留 1-2 秒上下文重叠。

#### 说明

这一阶段优先级低于 R1-R5。  
如果底层仍是逐帧流程，那么单独优化任务切分的收益有限；必须在轨迹级工作流建立后再做。

## 6. 推荐的下一步执行顺序

如果只按 ROI 排序，建议严格按下面顺序推进：

1. `R0` 已完成第一轮收稳，下一步直接进入 `R1`：补 `track_summary / risk_spans / issue_pool` 离线生成。
2. 再让 review 后端支持“下一问题点”派单。
3. 然后再做前端时间轴和问题点工作台。
4. 最后做关键帧传播和 QA 分层。

不建议的顺序：

1. 先花大量时间继续打磨逐帧右侧面板细节
2. 还没做风险摘要就尝试大规模人工提效
3. 还没做轨迹级工作流就先大动任务切分

## 8. 一句话总结

本项目的下一阶段目标不是“把逐帧修框做得更顺手一点”，而是把整个人工标注流程改造成：

- 轨迹级工作单位
- 关键帧修正 + 自动传播
- 风险片段驱动复核

现阶段已经完成“新数据按原始 IMU 段进入正式 batch + AI 预标注 + review 基线切换”，下一步最值得做的是：先把动态槽位 UI 收稳，再补离线风险摘要和问题派单能力。
