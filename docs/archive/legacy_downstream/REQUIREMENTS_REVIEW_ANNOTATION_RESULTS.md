# 需求文档 G：标注结果复审 UI（实现规划版）

## 1. 目标

本阶段目标：对已经产生的多标注结果做**二次复审**，让复审员在两份现有结果之间快速裁决，或在必要时重新绘制最终框。

交付目标：

1. 提供独立复审 UI，端口固定为 `10088`。
2. 每个待复审样本默认展示左右两张图，每张代表一种候选标注结果。
3. 复审员有三种选择：
   - 接受左边
   - 接受右边
   - 重新画
4. 点击左图或右图即可直接完成“接受左边/接受右边”。
5. 选择“重新画”后，进入一个类似现有标注 UI 的编辑界面。
6. 最终输出一份“复审后定稿结果”，供后续导出或继续分析。

---

## 2. 建议实现入口

建议新增：

1. 后端服务：`./codes/application/ui_review_result_server.py`
2. 前端页面：`./codes/application/ui_review_result_web/`

默认端口：`10088`

说明：

1. 当前仓库已经存在该复审服务实现入口，但它仍属于独立下游阶段，不是段模式主线入口。
2. 该服务与原始标注服务（`10086`）是不同阶段的独立 UI，不应直接覆盖原功能。

## 3. 输入范围与路径约束

主输入：

1. `./annotation/batch_<YYYYMMDD>_<vNN>/ui_tasks/ui_review.sqlite3`
2. `./data/required/<video_stem>/video/<video_stem>_retimed.mp4`
3. `./data/required/<video_stem>/video/<video_stem>_frame_timestamps_retimed.csv`
4. 原始复核结果：`annotations` 表
5. 可选分析辅助：`./annotation/batch_<YYYYMMDD>_<vNN>/annotation_analysis/*.dice_timeseries.csv`

路径规则：

1. 使用当前工作目录下相对路径（`./...`）。
2. 批次数据统一从 `./annotation/...` 读取。
3. 数据盘视频统一通过 `./data/...` 软链接访问。

---

## 4. 待复审样本定义

### 4.1 基本单位

一个待复审样本的单位是：

- `video_stem`
- `frame_index`

### 4.2 候选结果来源

每个样本默认只比较**两种候选结果**：

1. 左侧候选结果（left candidate）
2. 右侧候选结果（right candidate）

### 4.3 两个候选结果如何选取

强制规则：

1. 若某帧只有 2 条历史标注，则直接使用这 2 条。
2. 若某帧有 3 条及以上历史标注，则统一选择 **更相似的两条** 作为左右候选结果。
3. “更相似”建议按该帧综合 Dice 计算后，选择 Dice 更高的那一对。
4. 若存在多对并列，则使用稳定规则打破并列，例如：
   - 先按 Dice 降序
   - 再按 `submitted_at` 升序
   - 再按 `annotation_id` 升序
5. 候选结果的选取策略必须在日志与 summary 中写清。

---

## 5. 页面交互规范（强制）

### 5.1 主界面布局

每个待复审样本默认显示为左右双栏：

1. 左边：左候选结果
2. 右边：右候选结果
3. 顶部：样本信息（视频、帧号、时间戳、复审员 ID、当前进度、当前候选 Dice）
4. 底部或中部：操作区（重新画、跳过、语言切换等）

### 5.2 左右双图展示规则

左右两张图都应展示同一视频同一帧，但叠加不同的标注结果。

每张图中必须显示：

1. 原始视频帧
2. `P1` 框
3. `P2` 框
4. 来源说明（来自哪条历史 annotation）
5. 该候选结果与其对侧候选结果之间的当前 Dice 值

补充要求：

1. 左右候选结果的展示顺序应按当前 Dice **从小到大** 排列。
2. 当前 Dice 必须明确显示在页面上，不能要求复审员自行估计。

### 5.3 直接接受规则

复审员有两种“直接接受”动作：

1. 点击左图：表示“接受左边结果”
2. 点击右图：表示“接受右边结果”

要求：

1. 点击图片本体即可完成，不要求再点二次确认按钮。
2. 成功后立刻写入复审结果，并自动跳到下一帧。

### 5.4 重新画模式

若复审员认为左右两边都不合适，可以点击：

- `重新画`

点击后进入一个类似现有 `10086` 标注 UI 的编辑模式。

---

## 6. 重新画模式的交互规范（强制）

### 6.1 基本结构

重新画模式仍然是 `P1/P2` 双槽位编辑。

每个槽位必须支持三种选择：

1. 前人标注推荐框
2. 不存在（`absent`）
3. 手动画框

### 6.2 前人标注推荐框

这里的“推荐框”**不是 AI 推荐框**，而是：

- 来自左右两份候选结果中的现有框

例如对 `P1`：

1. 提供“采用左侧 P1 框”按钮
2. 提供“采用右侧 P1 框”按钮

对 `P2` 同理。

要求：

1. 推荐来源必须清楚标明是 `left` 还是 `right`。
2. 复审员点击后，可继续微调。

### 6.3 不存在

每个槽位都必须支持：

- `不存在`

规则：

1. 选择后写入 `bbox_x=y=w=h=0`
2. `source=absent`

### 6.4 手动画框

每个槽位都必须支持：

1. 鼠标拖拽绘制新框
2. 拖动与缩放已绘制框
3. 参数输入同步修改 `bbox_x/y/w/h`

### 6.5 来源字段

重新画模式下的最终来源建议允许以下枚举：

- `left_candidate`
- `right_candidate`
- `manual_draw`
- `manual_param`
- `absent`

## 7. 后端 API 规范（建议）

建议提供以下接口：

### 7.1 `POST /api/review_next`

返回下一个待复审样本。

返回内容至少包含：

1. `video_stem`
2. `frame_index`
3. `timestamp_ms`
4. 左候选结果
5. 右候选结果
6. 当前候选 Dice
7. 帧图地址

### 7.2 `POST /api/review_accept`

用于直接接受左右其中一侧。

请求至少包含：

1. `reviewer_id`
2. `video_stem`
3. `frame_index`
4. `accepted_side`（`left` / `right`）

### 7.3 `POST /api/review_redraw`

用于提交重新画后的最终结果。

请求至少包含：

1. `reviewer_id`
2. `video_stem`
3. `frame_index`
4. `p1_*`
5. `p2_*`
6. `final_source` 信息

### 7.4 `GET /api/frame_image`

沿用现有帧图获取方式即可。

## 8. 持久化建议

建议新增独立存储，不污染原始 `annotations` 表。

例如新增表：`review_decisions`

字段建议：

1. `decision_id`
2. `video_stem`
3. `frame_index`
4. `timestamp_ms`
5. `reviewer_id`
6. `reviewed_at`
7. `decision_type`（`accept_left` / `accept_right` / `redraw`）
8. `left_annotation_id`
9. `right_annotation_id`
10. `p1_bbox_x/y/w/h`
11. `p1_source`
12. `p2_bbox_x/y/w/h`
13. `p2_source`

说明：

1. 即使是 `accept_left/accept_right`，也建议把最终落定的 `P1/P2` 框冗余写入，避免后续查表再回溯。
2. 原始 `annotations` 表保持不变。

建议导出：

```text
./annotation/batch_<YYYYMMDD>_<vNN>/
├── review_results/
│   ├── final_reviewed.csv
│   └── review_decisions.jsonl
```

1. `final_reviewed.csv` 表示复审后的最终定稿结果
2. `review_decisions.jsonl` 表示每次复审动作明细

## 10. 验收标准

全部满足才算通过：

1. 服务可在 `10088` 端口启动。
2. 每个样本默认显示左右两张候选图。
3. 点击左图可直接接受左边并进入下一样本。
4. 点击右图可直接接受右边并进入下一样本。
5. 点击“重新画”可进入双槽位编辑界面。
6. 重新画界面支持：
   - 左候选推荐框
   - 右候选推荐框
   - 不存在
   - 手动画框
7. 最终结果独立落盘，不覆盖原始 `annotations`。

---

## 11. 失败处理规则

1. 若某帧无法构造左右候选结果，则该帧记为 `blocked` 并写日志。
2. 若点击接受后写库失败，不得 silently fail，必须提示并停留当前样本。
3. 若重新画结果非法（例如 `w<=0` 且非 absent），必须拒绝提交。
4. 若候选来源选择策略不明确，不允许默认“随便挑两条”。
5. 若页面未显示当前 Dice 或展示顺序不是按 Dice 从小到大，视为不满足需求。

---

## 12. 完成定义（DoD）

当且仅当以下条件全部满足：

1. `10088` 端口的复审 UI 可用。
2. 复审员可以在左右候选之间快速裁决。
3. 复审员可以在必要时进入重新画模式完成最终定稿。
4. 复审结果以独立结构保存，可导出。
5. 原始多标注结果仍然保留，复审结果可追溯到左右候选来源。
